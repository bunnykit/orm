# Type Generation

Type generation introspects your database schema and emits TypeScript declaration files for every table. The result is full IntelliSense (attribute access, column autocomplete in `where()`, type-checked `create()`) on **hand-written** model classes — no boilerplate added to your sources, no codegen running at runtime.

You don't have to use type generation. If you prefer to maintain attribute interfaces by hand, [`Model.define<T>()`](./typescript.md) gives the same IntelliSense from a TypeScript-only source. Pick whichever fits your workflow:

| Approach | Pros | Cons |
|---|---|---|
| `Model.define<T>()` | No build step, no generated files, types match exactly what you declare. | You re-declare attributes in two places (DB and TS). |
| Type generation | Single source of truth (the database). Auto-regenerates on every migration. | Adds a build step; generated files must be in `tsconfig.json`. |

```ts
const user = await User.first();
user.name;                      // ✓ autocomplete + type-checked
user.email = "a@example.com";   // ✓ typed setter
```

## Generating declarations

Run from the CLI:

```bash
# Generate alongside each model directory (preferred — uses modelsPath)
bunx bunny types:generate

# Custom output directory
bunx bunny types:generate ./src/generated

# Multi-tenant: landlord scope only
bunx bunny types:generate --landlord

# Multi-tenant: one specific tenant only
bunx bunny types:generate --tenant acme
```

You usually don't need to run this by hand. If `typesOutDir` (or `modelsPath`) is set, the migrator regenerates types automatically after every `migrate` and `migrate:rollback` — see [Migrations — Auto type generation](./migrations.md#auto-type-generation).

## Configuration

```ts
// bunny.config.ts
export default {
  connection: { url: "sqlite://app.db" },
  migrationsPath: ["./database/migrations", "./database/tenant-migrations"],

  // Per-model-root output (recommended)
  modelsPath: ["./src/models", "./src/admin/models"],

  // Or a single legacy output directory
  typesOutDir: "./src/generated/model-types",

  // Module specifier prefix used in generated `declare module` blocks
  typeDeclarationImportPrefix: "$models",

  // Whether to emit `class Foos { }` stubs (false) or augment your real class (true)
  typeStubs: false,

  // Override class names per table for irregular plurals or moved files
  typeDeclarations: {
    admin_users: { path: "$models/admin/AdminAccount", className: "AdminAccount" },
  },

  // Whether to derive class names by singularizing table names
  typeDeclarationSingularModels: true,
};
```

### `modelsPath` placement

With `modelsPath` set, Bunny writes `types/` beside each model root:

```
src/
├── models/
│   ├── User.ts                 ← your hand-written model
│   ├── BlogPost.ts
│   └── types/                  ← generated by bunny
│       ├── users.d.ts
│       └── blog_posts.d.ts
└── admin/
    └── models/
        ├── AdminAccount.ts
        └── types/
            └── admin_users.d.ts
```

This keeps generated declarations close to the files they augment. Make sure your `tsconfig.json` includes them:

```jsonc
{
  "include": ["src/**/*.ts", "src/**/*.d.ts"]
}
```

If you set `typesOutDir` instead, all `.d.ts` files land in one directory — convenient for `.gitignore` but requires a working module specifier from the consumer's path.

### Module specifiers

Bunny reads `tsconfig.json`'s `compilerOptions.paths` and emits a `declare module` block for each alias the model file resolves to. That means autocomplete works regardless of how you import:

| Table | Model file | No prefix (relative) | With `$models` prefix |
|---|---|---|---|
| `users` | `src/models/User.ts` | `../User` / `User` | `$models/User` / `User` |
| `blog_posts` | `src/models/BlogPost.ts` | `../BlogPost` / `BlogPost` | `$models/BlogPost` / `BlogPost` |
| `tenants` | `src/models/landlord/tenant.ts` | `../landlord/tenant` / `Tenant` | `$models/landlord/tenant` / `Tenant` |

### Singular vs plural model names

By default, Bunny derives class names by singularizing table names (`users` → `User`, `blog_posts` → `BlogPost`). Set `typeDeclarationSingularModels: false` if your models use plural names (`class Users`).

For irregular cases (`media`, `criteria`, `data`), use `typeDeclarations` to map a table to an exact module path and class name:

```ts
typeDeclarations: {
  media: { path: "$models/Media", className: "Media" },
  criteria: { path: "$models/Criterion", className: "Criterion" },
},
```

## Multi-tenant generation

For multi-tenant apps, partition `modelsPath` into scopes:

```ts
export default {
  connection: { url: "postgres://localhost/landlord" },
  modelsPath: {
    landlord: "./src/models/landlord",
    tenant:   "./src/models/tenant",
  },
  typesOutDir: "./src/generated/types",
  typeDeclarationImportPrefix: "$models",
  tenancy: {
    resolveTenant: (tenantId) => ({
      strategy: "schema",
      name: "tenant_db",
      schema: `tenant_${tenantId}`,
    }),
    listTenants: () => ["acme", "globex"],
  },
};
```

Bunny introspects each scope against the right connection:

- **Landlord** types — generated from the default connection.
- **Tenant** types — generated by resolving one representative tenant (via `TenantContext.run`). Because all tenants share the same schema, one is enough.

```bash
bunx bunny types:generate              # both scopes
bunx bunny types:generate --landlord
bunx bunny types:generate --tenant acme
```

> **Tip:** Keep landlord and tenant model directories non-overlapping. If one scope's path is nested inside the other (e.g. `tenant: "./src/models"` containing `landlord/` inside it), Bunny excludes the nested scope from discovery so types don't leak across scopes.

## What gets emitted

For each table, Bunny generates an `Attributes` interface plus (optionally) a `declare module` block that merges those attributes into your real model class:

```ts
// src/models/types/users.d.ts
import { User } from "../User";

export interface UsersAttributes {
  id: number;
  name: string;
  email: string | null;
  created_at: string;
}

declare module "../User" {
  interface User extends UsersAttributes {}
}
```

Your model stays hand-written:

```ts
// src/models/User.ts
import { Model } from "@bunnykit/orm";

export class User extends Model {
  static table = "users";

  posts() {
    return this.hasMany(Post);
  }
}
```

With both files in place:

```ts
await User.where("email", "alice@example.com").first();   // ✓ column autocompletes
await User.create({ id: 1, name: "Alice", email: "a@b" }); // ✓ payload type-checked
const user = await User.first();
user!.name;                                                // ✓ string
```

Column arguments still accept raw strings for joins, aliases, expressions, and SQL fragments, so the autocomplete doesn't block escape hatches.

### Stubs mode

If you'd rather Bunny generate the base classes themselves (instead of augmenting hand-written ones), set `typeStubs: true`. The generator then emits:

```ts
// src/generated/model-types/User.ts
export class User extends Model<UsersAttributes> {
  static table = "users";
}
```

You extend these stubs in your own files. This is closer to the Prisma-style "generated client" pattern. Stubs are exposed for projects that prefer that style; the augmentation approach above is the default because it keeps your model source files canonical.

## Programmatic generation

```ts
import { TypeGenerator, Connection } from "@bunnykit/orm";

const connection = new Connection({ url: "sqlite://app.db" });
const generator = new TypeGenerator({
  connection,
  outDir: "./src/generated/model-types",
  modelDirectory: "./src/models",
  modelImportPrefix: "$models",
  singularModels: true,
  declarations: true,
  stubs: false,
});

await generator.run();
```

The `configureBunny()` facade and the `bunny types:generate` CLI both wrap this.

## Workflow

The typical cycle is:

1. Write a migration with `bunny migrate:make`.
2. Run `bunx bunny migrate` — schema changes and types regenerate.
3. Use the new column / relation in your code; IntelliSense already knows about it.

You can safely add the generated directory to `.gitignore` if you regenerate as part of `bun install` or CI — types are pure derivatives of your schema. Keeping them committed is also fine if you want commit history to track schema changes.

## Common pitfalls

- **Types not picked up by the editor.** Ensure your `tsconfig.json` `include` covers the directory holding `.d.ts` files. `src/**/*.ts` does **not** include `.d.ts` unless you also list `**/*.d.ts`.
- **Wrong model name from irregular table.** Set `typeDeclarationSingularModels: false` or add a `typeDeclarations` override.
- **Generated files re-augment a missing class.** If you delete a model but leave the table around, the next regeneration emits a `declare module "../User"` that points nowhere. Delete the corresponding `.d.ts` (`types/users.d.ts`) too.
- **Tenant types out of sync.** If only one tenant has run new migrations, the introspected schema may not match other tenants. Run tenant migrations across every tenant before regenerating.
